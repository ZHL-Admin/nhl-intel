(`ls models_ml/*.py | wc -l` → `73`), including `__init__.py`. I inventory all 73. The 75 figure
may count the 2 sub-package files elsewhere or predate 2 deletions; I could not reconcile it and
flag it as an unresolved discrepancy, not an error in this inventory.

### How "serving_tables.yml reference" is interpreted (uncertainty flagged)
`serving_tables.yml` names **tables**, never scripts. I therefore record a serving_tables.yml
"reference" for a script when the script's **output table** appears in that manifest (evidence the
table is materialized + served). Where a script is reachable ONLY through its output-table entry
(never named by module in Makefile/DAGs/precompute/Handoff), I say so explicitly.

## 2. Duplication-smell resolutions

### `archetype_features.py` (v1) vs `archetype_features_v2.py`
**Both are LIVE; neither is superseded-and-dead.**
- v1 `archetype_features.py` is imported by `linefit_features.py`
  (`models_ml/linefit_features.py:26: from models_ml import archetype_features, bq, config`) and by
  v1 `fit_archetypes.py:34`. `linefit_features` is reachable from the six sources
  (`precompute_serving.py:32 import … linefit_features as lf`; `train_linefit`; `score_line`;
  `backend/services/tools.py`), so the **v1 feature module rides the live line-fit path** and is
  NOT dead.
- v2 `archetype_features_v2.py` is imported by `fit_archetypes_v2.py:36` and
  `compute_archetype_explainer.py:29,31` — both live. LIVE.
- Verdict: they serve **different consumers** (v1 → line-fit features; v2 → archetype clustering +
  explainer). The "v2 supersedes the v1 vector" note in the v2 docstring refers to the
  **archetype-clustering** vector only, not the line-fit feature module. No deletion candidate here.

### `fit_archetypes.py` (v1) vs `fit_archetypes_v2.py`
**v1 `fit_archetypes.py` is SUPERSEDED and reachable by none of the six** (candidate-with-ambiguity, §5).
- Deciding references: `dags/nhl_daily.py:589-598` runs `fit_archetypes_v2 --write` (task
  `write_archetypes`); `Makefile:135` (`archetypes-v2`) runs `fit_archetypes_v2 --write`. Neither
  the DAGs, Makefile, precompute, serving_tables, nor Handoff name `fit_archetypes` (v1).
- The v2 docstring is explicit: `fit_archetypes_v2.py:2 "Archetype refit v2 (supersedes
  fit_archetypes.py / archetypes_v1)."`
- Both write the SAME table `nhl_models.player_archetypes`, but with different `model_version`:
  v1 `fit_archetypes.py:194 out["model_version"]="archetypes_v1"`; v2
  `fit_archetypes_v2.py:323 out["model_version"]="archetypes_v2"`. The live pipeline runs v2, so the
  served `player_archetypes` carries `archetypes_v2` — confirmed downstream by
  `project_roster_forecast.py:496 WHERE … model_version = 'archetypes_v2'` and by
  `compute_archetype_explainer` reading the v2 artifacts.

  committed, canonical GMM (archetypes_v1.joblib)", but the command it runs is `fit_archetypes_v2`,

  "archetypes_v2.joblib"`). The comment is stale; the executed code uses v2. Flagged, not resolved
