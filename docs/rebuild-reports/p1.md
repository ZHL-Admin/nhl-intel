# Rebuild P1 — dbt rebuild of the shift backbone

Branch `rebuild/dedup-segments-retrain`. All model changes reversible via dbt (no
manual table surgery). `research/deployment-atlas/` used read-only as the
validation reference.

## Changes made
1. **Exact-duplicate shift drop at staging** (`models/staging/stg_shifts.sql`):
   `qualify row_number() over (partition by game_id, player_id, period,
   shift_start_seconds, shift_end_seconds order by shift_number) = 1`. Matches the
   Atlas rule (identical shifts collapse to one).
2. **`int_shift_segments` rebuilt with goal cut-points** (`models/intermediate/
   int_shift_segments.sql`): added a `goals` CTE (non-SO goals, `(period-1)*1200 +
   mm:ss`) unioned into the boundary set, so **score state is constant per segment**.
   All other conventions preserved (strength from the ice, `positionCode='G'`
   goalie ID, `(period-1)*1200` offsets, actual OT lengths, `≤6 skaters/side`).
3. **`int_segment_context`, `int_on_ice_events`** rebuilt on top (no logic change).
4. **Marts refreshed** on the new backbone: `mart_player_onice` (15.8k),
   `mart_player_toi_matrix` (209.5k), `mart_player_wowy` (418.6k), plus their
   intermediates (`int_segment_5v5_results` 5.3M, `int_player_onice_game` 735k).

dbt runs: `PASS=4` (backbone) then `PASS=12` (marts + deps), 0 errors.

## AGAINST relation on `mart_player_toi_matrix` — NOT forced (grain does not permit)
Its grain is `(season, team_id, player_id_a, player_id_b)` — **teammate-specific**
(both players share one `team_id`). Opponents span two teams, so an AGAINST
relation does not fit this grain without either dropping `team_id` or adding
`team_id_a/team_id_b` + a `relation` column — a schema change to a mart with
existing consumers. **Per the work order, saying so rather than forcing it.**
Recommendation: a **sibling model** `mart_player_toi_against` (grain `season,
player_id_a, player_id_b, toi_against_sec`) if the AGAINST matrix is needed in the
warehouse; the Atlas already exposes it via `api.shared_toi(..., relation="against")`.

## Staleness — operational, not structural
The daily DAG (`dags/nhl_daily.py`) **already rebuilds these nightly**:
`run_dbt_pre_xg` (`dbt run --exclude path:models/mart …`) builds `int_shift_segments`
+ context + on-ice-events; `run_dbt_marts` (`dbt run --select path:models/mart …`)
builds the marts. Both are `table` materializations, fully rebuilt each run. The
tables sat at 2026-07-02/06 because the **Airflow scheduler was not running the DAG**
(dev environment) — an *operational* gap, and those builds predate the
backfill+dedup work. **No structural dbt change is needed** — the models are wired;
the P1 SQL fixes will take effect automatically on the next scheduled run.
Recommendation: confirm the scheduler runs the daily DAG (or add a cadence'd
`dbt build`); optionally split the heavy `int_shift_segments` into its own task for
observability.

**[1] detail (per the boundary rule — reported, not tuned):** on the correct
apples-to-apples basis (both sides drop impossible-personnel `>6/side` stints —
production via `≤6 skaters/side`, the Atlas via its `is_quarantined` flag), **19,139
of 19,149 games match within 2s (99.95%)**. The remaining **10 games** are **corrupt
period-5-shift anomalies** — a handful of games carry on-ice shifts labelled period 5
(shootout period; real value should be none), e.g. `2021020972` (838 shifts, 3 in
"period 5", shifts identical in both pipelines). The Atlas extends its stint span to
the period-5 boundary (6000s); the rebuilt production segments truncate earlier.
Both handle the corrupt input defensibly-but-differently. These 10 games (0.05%) are
a **data-anomaly tail, not a rebuild error** — the shifts are byte-identical between
pipelines and every analytical check ([2]/[3]/[4]) passes cleanly.

Note: without the apples-to-apples basis, 172 games differed — that gap was entirely
the **quarantine-handling difference** (Atlas flags-and-keeps its 753 impossible
stints in the raw per-game total; production drops them). The 5v5 numbers are
unaffected (those stints aren't 5v5), which is why [2]/[3] reconcile exactly.
